==========================================================
    BACKEND APPLICATION FLOW (SPRING BOOT REST API)
==========================================================

This document explains the architecture and data flow of the backend 
for the Client Project Dashboard application. The application follows 
the industry-standard **N-Tier Architecture**, which cleanly separates 
concerns into different layers.

----------------------------------------------------------
1. THE CORE LAYERS
----------------------------------------------------------
The system is divided into 4 primary layers, represented by the packages 
in `src/main/java/com/dashboard/`:

A. Model Layer (`com.dashboard.model`)
   - Also known as the Domain layer or Presentation layer.
   - Contains JPA Entities (e.g., `Project.java`).
   - These classes directly map to database tables. Every instance 
     represents a row in the table, and fields represent columns.

B. Repository Layer (`com.dashboard.repository`)
   - Also known as the Data Access Layer (DAL).
   - Contains interfaces (e.g., `ProjectRepository.java`) that extend Spring 
     Data JPA (like `JpaRepository`).
   - Handles all database interactions (CRUD operations: Create, Read, Update, Delete) 
     without requiring boilerplate SQL queries. Spring automatically generates 
     the SQL queries based on method names.

C. Service Layer (`com.dashboard.service`)
   - The heart of the application ŌĆō the Business Logic Layer.
   - Contains classes/interfaces (e.g., `ProjectService.java`) annotated with `@Service`.
   - Where the "rules" of your business live (validation, complex processing, 
     calling external APIs, integrating multiple repositories).
   - The Controller calls the Service, and the Service calls the Repository.

D. Controller Layer (`com.dashboard.controller`)
   - Also known as the Presentation or API Layer.
   - Contains classes annotated with `@RestController` (e.g., `ProjectController.java`).
   - Acts as the entry point for HTTP requests from the frontend (React/Angular).
   - Its ONLY job is to:
      1. Accept the HTTP request (GET, POST, PUT, DELETE).
      2. Parse the request body/parameters.
      3. Call the appropriate Service method.
      4. Take the result from the Service and format it as an HTTP Response (usually JSON).

----------------------------------------------------------
3. HOW THE COMPONENTS CONNECT (Dependency Injection)
----------------------------------------------------------
Spring Boot uses a design pattern called Dependency Injection (DI) to connect 
these layers automatically:
- The Controller doesn't create a new Service using `new ProjectService()`.
- Instead, Spring "injects" the Service into the Controller automatically.
- This is typically done via Constructor Injection or the `@Autowired` annotation.

----------------------------------------------------------
4. THE ENTRY POINT
----------------------------------------------------------
`DashboardApplication.java`
- Contains the `main()` method.
- Annotated with `@SpringBootApplication`.
- When you run this file, Spring Boot starts the embedded web server (Tomcat), 
  scans for all your models/controllers/services/repositories, connects to the 
  H2 database, and listens for HTTP traffic.
